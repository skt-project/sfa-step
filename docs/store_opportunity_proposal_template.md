# STEP — Offer Proposal Template Design

This document defines the layout, typography, field bindings, and print/PDF generation strategy for the Offer Proposal produced by the Store Opportunity module (Page 6).

---

## 2. Field Bindings

| Template field | Source |
|---|---|
| `[LOGO]` | Configurable per distributor — a GCS-hosted image URL stored in a `distributor_profile` config (to be added to Master Data module); fallback to Skintific/G2G default logo |
| `{proposal_number}` | `proposal_header.proposal_number` (e.g. `PROP/DST171/202607/0001`) |
| `{proposal_date}` | `proposal_header.proposal_date` formatted as `DD MMMM YYYY` (Indonesian locale, e.g. "01 Juli 2026") |
| `{valid_until}` | `proposal_header.valid_until` same format |
| `{store_name}`, `{store_code}`, `{region}`, `{channel}` | `proposal_header` fields copied at generation time — NOT re-queried at print time, so a store's details remain frozen as of when the proposal was generated |
| `{salesman_name}`, `{salesman_code}` | `proposal_header` fields |
| Item table | `proposal_item` rows, ordered by `item_order ASC` |
| `Hist. Demand` | `proposal_item.historical_demand` |
| `Avg/Mgg` | `proposal_item.weekly_avg_demand` |
| `Rekomendasi` | `proposal_item.recommended_quantity` — the final value confirmed by the Distributor Admin, not necessarily the system suggestion |
| `{total_sku_count}`, `{total_quantity}` | `proposal_header.total_sku_count`, `.total_quantity` |
| `{notes}` | `proposal_header.notes` (blank if not provided) |
| `{generated_by}` | `proposal_header.generated_by` (STEP user display name) |
| Footer signature blocks | Empty fields for physical signature at printing |

## 3. Typography and Style

| Element | Specification |
|---|---|
| Page size | A4 (210mm × 297mm) |
| Margins | 20mm top/bottom, 15mm left/right |
| Header font | Bold, 11pt, distributor brand color (configurable, default #0066E5 matching STEP palette) |
| Body font | Regular, 9pt, #10202F (STEP --ink color) |
| Table header | Bold, 9pt, white on #0066E5 background |
| Table rows | Alternating #FFFFFF / #F6F9FC (STEP --soft-white) |
| Footer | Italic, 8pt, #54677A (STEP --ink-soft) |
| Branding | "Generated by STEP · {date} {time}" bottom-right corner, 7pt |

---

## 4. Multi-Page Handling

When `proposal_item` rows exceed one page (~20 items at default font size), the template repeats the header block (distributor info + store info) on continuation pages as a sticky header, with "Hal X / N" (page X of N) printed top-right. The footer (signatures + T&Cs) appears ONLY on the final page.

---

## 5. PDF Generation

Server-side generation using **Puppeteer** (headless Chromium) or **wkhtmltopdf** — the template is a server-rendered HTML string (Jinja2 or equivalent server-side template), not a browser-rendered screen. Steps:
1. API loads `proposal_header` + `proposal_item` from BigQuery.
2. Server-side renders HTML template with field bindings.
3. Puppeteer/wkhtmltopdf renders HTML → PDF, returning a byte stream.
4. `Content-Disposition: attachment` header with filename `{proposal_number}.pdf`.
5. Audit record written to `proposal_history`.

**Why server-side, not browser print:** distributor logos and brand colors are resolved server-side, keeping credentials/paths off the client. Also allows reprinting the EXACT same PDF (same data snapshot from `proposal_header`/`proposal_item`) regardless of whether underlying demand data has since changed.

---

## 6. Future: QR Code

The template includes a placeholder (`[QR_AREA]` in the top-right corner) for an optional QR code. When implemented, it would encode: `proposal_id + proposal_number + distributor_code` — enabling a store owner to scan and verify the proposal's authenticity via a public STEP URL (`/verify/{proposal_id}`). Not built in this phase — the placeholder is reserved in the layout to avoid a template redesign later.
